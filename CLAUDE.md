# CLAUDE.md

## 1. 编码前先思考

**不要假设。不要隐藏困惑。要主动暴露权衡。**

实现之前：
- 明确陈述你的假设。如果不确定，请问。
- 如果存在多种理解方式，都要呈现出来——不要默默选择其中一个。
- 如果存在更简单的方案，请指出。有必要时要敢于反驳。
- 如果有不清楚的地方，停下来。说出哪里令人困惑，然后提问。

## 2. 简单优先

**最小化代码解决问题。不做过度设计。**

- 不要添加需求之外的功能。
- 不要为一次性使用的代码创建抽象。
- 不要添加未被要求的"灵活性"或"可配置性"。
- 不要为不可能发生的场景添加错误处理。
- 如果能写 50 行就别写 200 行，重写它。

问自己："一位资深工程师会说这太复杂了吗？"如果是，请简化。

## 3. 精准修改

**只改动必须改动的。只清理自己造成的混乱。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有损坏的部分。
- 保持现有风格一致，即使你会有不同的写法。
- 如果发现无关的死代码，提一下就行——不要删除它。

当你的修改产生了孤立代码时：
- 移除因你的修改而不再使用的 import/变量/函数。
- 不要删除之前就存在的死代码，除非被要求。

检验标准：每一行修改都应该能直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环验证直到完成。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让测试通过"
- "修复 bug" → "编写能复现问题的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，请先陈述简要计划：
```
1. [步骤] → 验证：[检查方法]
2. [步骤] → 验证：[检查方法]
3. [步骤] → 验证：[检查方法]
```

强有力的成功标准让你能够独立循环验证。弱标准（"让它能工作"）需要不断确认。

---

**这些准则生效的表现：** diff 中的不必要更改减少、因过度复杂导致的返工减少、以及问题在错误实现之前被澄清而非之后。
